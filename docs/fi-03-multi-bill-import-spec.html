FI-003 — Batch Bill Import
Workflow specification for importing multiple supplier invoices in one operation — whether dropped as separate files or contained within a single multi-invoice PDF.
📐 Spec — Ready to Build 🕐 Not Yet Built Last updated: Mar 2026 Owner: Paul Tyrrell
Overview Entry Points Upload Interaction Multi-Invoice Detection Processing Queue Batch Review Screen Matching Rules Bulk Apply API Changes Edge Cases Roadmap
Section 01
Overview
The Batch Bill Import workflow handles two scenarios that today require multiple separate uploads. Both converge on the same Batch Review Screen — a split-panel interface where every detected invoice is listed with its match status and the user can review, adjust, and apply them in bulk.
Design principle: Uploading ten bills should not be ten times the work. Anything the system can match automatically is pre-approved — the user's job is to resolve exceptions only.
ScenarioInputDetectionOutcome
A — Multiple Files User drops or selects 2–20 individual PDF / image files Each file parsed independently N invoices on the Batch Review Screen
B — Single Multi-Invoice PDF One PDF containing multiple concatenated supplier invoices (e.g. from accounting export, email attachment bundle) Claude detects multiple invoice sections; returns array N invoices from 1 file on the Batch Review Screen
C — Single File (existing) One PDF with one invoice (current behaviour) Array of length 1 returned Batch Review Screen with 1 item — or auto-route to Bill Detail
Section 02
Entry Points
Users reach the batch import flow from two places. The key change from the current design is that the Bill Detail page drops the PDF zone for single-file upload — that experience moves to the new Batch Import page which handles both single and multiple files gracefully.
A
Bills sidebar / nav item → "Import Bills" Frontend
New dedicated page at /bills/import. The primary entry point for all bill uploads — single or batch. Replaces the embedded drop zone on Bill Detail for new imports. Shows the full upload zone and batch queue.
B
Bill Detail page → "Import from PDF" button Frontend
The existing Bill Detail page keeps a compact "Import from PDF" button for users who arrive at a specific bill and want to parse it. This opens the same upload flow but pre-scoped to that bill's supplier (pre-fills the supplier filter in review).
Routing rule: If the batch result contains exactly 1 invoice AND it cleanly matches a known PO, the flow auto-routes to Bill Detail with the invoice pre-populated (no review screen needed — zero friction for the common single-bill case).

Section 04
Multi-Invoice Detection
The most powerful part of this workflow. When a single PDF contains multiple supplier invoices — common with email-batched attachments or accounting system exports — Claude detects the boundaries between them and returns each as a separate, fully-parsed invoice object.
Real-world trigger: A supplier emails a single PDF with 6 invoices for the month. Or a bookkeeper exports all unpaid bills from Xero as one document. Today, each invoice requires a separate upload. This spec eliminates that entirely.
Prompt Update — Invoice Schema Change
The existing /api/parse-invoice prompt is updated to always return an array of invoices. Single-invoice PDFs return an array of length 1. This makes the client logic uniform and eliminates a separate batch endpoint.

Prompt Instruction — Split Detection
Key prompt addition: "If the document contains more than one supplier invoice (identified by separate invoice numbers, dates, or supplier headers), return each as a separate object in the invoices array. Do not merge line items across separate invoices."
Server: Backwards Compatibility
The server normalises both old (single invoice object) and new (invoices array) shapes so the transition is non-breaking:
// server.js — normalise response shape
const parsed = JSON.parse(cleaned);
const invoices = Array.isArray(parsed.invoices)
  ? parsed.invoices
  : [parsed]; // legacy single-object response

res.json({ invoices }); // always returns { invoices: [...] }

Section 07
Per-Invoice Matching Rules
Each invoice goes through the same matching logic as the existing single-bill flow. A 2% tolerance catches rounding and minor freight variations without flagging everything as an exception.
1
PO match on customerPoRef
Compare the invoice's customerPoRef (our PO number as printed on the invoice) against all open FieldInsight PO IDs. Case-insensitive exact match.
pos.find(p => p.id.toLowerCase() === invoice.customerPoRef.toLowerCase())
2
Variance check → Matched or Needs Review
If a PO is found: compare invoice.totalIncGst to po.totalValue. If variance ≤ 2% of PO value → Matched. If variance > 2% → Needs Review.
const variance = Math.abs(invoice.totalIncGst - po.totalValue); const tolerance = po.totalValue * 0.02; const status = variance <= tolerance ? 'matched' : 'differs';
3
No match → No PO Found
No PO with a matching ID found. The invoice is flagged. User options: search and link to an existing PO, create a new PO, or skip the invoice. Unlike statements, invoice matching is PO-ID only — no tiered project/job fallback (the PO number should always be on an invoice).
4
Duplicate detection
Before adding to the review list, check if invoice.supplierInvoiceNo + invoice.supplierName combination already exists in the bill register. If so, flag as Possible Duplicate with a warning — user must confirm before syncing.
Section 08
Bulk Apply Flow
The power of batch import is in bulk approval. "Apply All Matched" processes every green invoice in sequence, syncing each to its PO with a single click. The user sees a live counter and a final summary.
1
"Apply All Matched (N)" clicked User Action
A confirmation modal lists the N invoices that will be synced, their POs, and the total dollar value being applied. "Confirm Apply" proceeds.
2
Sequential apply with live counter Frontend
Invoices are applied one at a time (not in parallel) to avoid race conditions on shared POs. A progress indicator shows "Applying 2 of 3…". Each invoice calls the existing updatePurchaseOrder / addPOLineDetails DataContext functions.
// Per matched invoice — existing sync logic for (const inv of matchedInvoices) { await syncBillToPO(inv.matchedPoId, inv); setAppliedCount(n => n + 1); }
3
Summary screen User Feedback
Once all matched invoices are applied, a summary card appears: "3 bills synced · 2 need your attention". The review list auto-filters to show only the unresolved items. The user can then work through exceptions individually.
4
Individual exception resolution User Action
Each "Needs Review" invoice shows the variance amount and offers: Add Line to PO (adds a MISC/freight line for the difference), Accept & Sync (applies despite variance — for known discrepancies), or Skip (defer to later). Each "No PO" invoice offers: Search & Link, Create PO, or Skip.

Section 10
Edge Cases
Common scenarios that need explicit handling so the user is never stuck.
ScenarioExpected Behaviour
Same PO matched by 2+ invoices Both shown in review. User applies first one — second now shows the updated PO value, variance recalculated. If second also matches within tolerance, it remains green. User is warned: "PO-1648 has already had a bill synced in this batch."
Invoice already in system (duplicate) Flagged as "Possible Duplicate" with amber warning. Shows the existing bill record. User must explicitly choose "Apply anyway" or "Skip". Never silently overwrites.
PDF parse returns 0 invoices File shows as Error in queue. Error panel: "Claude couldn't find any invoice data in this file." Retry button + manual skip.
One file in a 10-file batch fails The other 9 continue and appear in review normally. Failed file shown in queue as red with "Retry" button. User can proceed with the 9 and retry or skip the failed one separately.
Multi-invoice PDF but same supplier + same PO across all invoices Each invoice is still a separate row in the review list. They may all match the same PO. User can bulk-apply them — total will exceed PO value, so second and third will show variance. Designed behaviour: the PO was only for one delivery; user must create PO for subsequent ones.
File dropped onto existing Bill Detail page Bill Detail retains a compact "Import from PDF" button. Clicking it opens the Batch Import flow at /bills/import and pre-fills the supplier selector with the current bill's supplier.
20+ files dropped at once Cap at 20 files per batch with a clear message: "Batch limit is 20 files. The extra N files were not added." Prevents excessively long queues and API overload.
ANTHROPIC_API_KEY not set All files in queue immediately move to "Demo mode" status. Review screen pre-populates with seeded demo invoices per-supplier. Blue info callout: "API key not configured — showing demo data."
