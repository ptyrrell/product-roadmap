Section 01
Overview
FieldInsight processes two types of supplier PDF documents using Claude Vision AI. Both share the same server-side architecture but use different prompts and client-side matching logic.
Document Type Endpoint Purpose Client Output
Supplier Invoice / Bill /api/parse-invoice Extract line items for bill reconciliation against a PO Line items matched to product catalogue
Supplier Statement /api/parse-statement Extract PO references + amounts for bulk statement reconciliation Rows matched to FieldInsight POs by tier
Design principle: Parsing runs server-side (Express) so the Anthropic API key is never exposed to the browser. The React app calls /api/parse-invoice or /api/parse-statement — the server handles the Anthropic call and returns clean JSON.

Section 03
Invoice / Bill Parser
Extracts all line items from a supplier invoice. Called when a user drops a PDF onto the Bill Detail page. Powers the bill-to-PO reconciliation workflow.
1
User drops PDF on Bill Detail page Frontend
File is immediately posted to the server while the progress stepper animates through 5 steps (900ms intervals). The PDF iframe preview loads in parallel.
POST /api/parse-invoice Content-Type: multipart/form-data Body: file = <PDF or image, max 25MB>
2
Server sends to Claude Vision Backend
File buffer is base64-encoded and wrapped in an Anthropic document block (PDF) or image block (JPEG/PNG). The invoice extraction prompt is appended as a text block.
// PDF document block { type: 'document', source: { type: 'base64', media_type: 'application/pdf', data: base64 } } // Image block { type: 'image', source: { type: 'base64', media_type: 'image/jpeg', data: base64 } }
3
Claude extracts line items Claude AI
Claude reads the document and returns a strict JSON object. The prompt instructs it to act as an accounts-payable clerk. Key rules: lineTotal is always ex-GST; unitPrice matches gstTreatment; missing supplier codes become slugs like MISC-001.
4
Server cleans + returns JSON Backend
Strips accidental markdown fences, parses JSON, adds id, internalCode: '', and codeMatch: 'none' stubs to each line. Returns the enriched array.
5
Client matches codes + auto-populates Matching
Browser runs matchCodes() — compares each supplierCode against the product catalogue. Match quality stored as exact | partial | none. customerPoRef auto-fills the Bill's Reference field. documentGst sets the GST toggle.
Invoice Response Schema
{
  "supplierInvoiceNo": "string",
  "invoiceDate":       "YYYY-MM-DD",
  "supplierName":      "string",
  "customerPoRef":     "string  — customer's own PO number as printed on invoice",
  "documentGst":       "ex" | "inc",
  "subtotalExGst":     number,
  "totalGst":          number,
  "totalIncGst":       number,
  "lines": [
    {
      "supplierCode":  "string  — supplier's SKU / part number",
      "description":   "string",
      "qty":           number,
      "unitPrice":     number  // matches gstTreatment,
      "gstTreatment":  "ex" | "inc",
      "gstAmount":     number  // lineTotal × 0.1, or 0 for GST-free,
      "lineTotal":     number  // ALWAYS ex-GST
    }
  ]
}
Section 04
Statement Parser
Extracts PO references and amounts from a supplier account statement. Called when a user drops a statement PDF on the Statement Import page. Enables bulk reconciliation of multiple POs at once.
Key distinction — two reference fields per line. A supplier statement records two different reference numbers for each transaction. The parser extracts both and the matching logic uses them differently.
FieldDescriptionStatement ColumnExample
supplierPoRef The customer's PO number the supplier recorded when the order was placed "Purchase Order No.", "Customer PO", "Your Order No." PO-1648
supplierRef The supplier's own internal reference for this transaction "Reference", "Doc No.", "Job No.", "Docket" DAIKIN-JOB-99234
Statement Response Schema
{
  "supplierName":   "string",
  "statementDate": "YYYY-MM-DD",
  "statementRef":  "string  — statement or account number",
  "totalIncGst":   number,
  "lines": [
    {
      "supplierPoRef": "string  — customer's PO number as recorded by supplier",
      "supplierRef":   "string  — supplier's own job / docket / invoice ref",
      "amountIncGst":  number  // positive, inc GST,
      "description":   "string"
    }
  ]
}
Prompt Rules
  • Skip header rows, sub-total rows, opening/closing balance rows
  • Use individual transaction amounts — not running balance totals
  • All amounts must be positive numbers inc GST
  • Returns {"error":"Not a supplier statement"} if document type is wrong
  • Auto-fills Supplier, Statement date, and Statement reference from parsed metadata
Section 05
Matching Logic
Client-side matching runs after the parse response arrives. For statements, matching is tiered — falling through to progressively broader match criteria before declaring "No PO Found".
Statement — Tiered Match
T1
Exact PO ID match Tier 1
supplierPoRef compared against FieldInsight PO id (case-insensitive exact match). This is the primary match — works when the supplier correctly records your PO number.
pos.find(p => p.id.toLowerCase() === supplierPoRef.toLowerCase())
T2
Project reference match Tier 2
supplierPoRef compared against PO reference field (the project reference). Used when the supplier records your project code instead of your PO number. Status badge shows "via Project #".
pos.find(p => p.reference?.toLowerCase() === supplierPoRef.toLowerCase())
T3
Job reference match Tier 3
supplierPoRef compared against PO jobRef field. Used when the supplier records your job number. Status badge shows "via Job #".
pos.find(p => p.jobRef?.toLowerCase() === supplierPoRef.toLowerCase())
—
No PO Found
No match across all three tiers. The supplierPoRef is shown in purple in the Statement PO column. Actions available: Create PO, Add new PO, + Stmt Amt to PO Line.
Match Statuses
StatusConditionToleranceBulk-actionableActions
Matched PO found, variance within tolerance ±2% ✅ Yes Mark Received
Set from $0 PO found, PO value is $0 n/a ✅ Yes Set value + Mark Received
Needs Review PO found, variance exceeds tolerance >2% ❌ No Review on PO · + Stmt Amt to PO Line
No PO Found No match across all three tiers n/a ❌ No Create PO · Add new PO · + Stmt Amt to PO Line
"+ Stmt Amt to PO Line" Action
When to use: A statement line can't be cleanly matched (or has an amount difference), but it belongs to an existing FieldInsight PO. Instead of creating a new PO, add the supplier charge as a MISC line on the existing PO — preserving the supplier's reference in the description for audit trail. 
// Line added to the selected FieldInsight PO
{
  poId:           "PO-1648",          // selected by user
  inventoryCode:  "MISC",             // default, user can change
  description:    "Supplier Ref: DAIKIN-JOB-99234 — description",
  orderedQty:     1,
  receivedToDate: 1,                 // auto-marked received
  unitPrice:      290.64,            // amountIncGst / 1.1
  discount:       0
}

Section 07
Error Handling
All parse failures fall back to demo mode — the user always sees something useful even when the API is unavailable.
ScenarioHTTP StatusClient Behaviour
ANTHROPIC_API_KEY not configured 503 Demo mode — pre-seeded items shown, blue info alert: "API key not configured"
Claude returns non-JSON text 422 Warning alert + demo fallback
Claude flags wrong document type 422 {"error":"..."} Warning alert message shown + demo fallback
File too large (>25 MB) 400 Error alert — no fallback, user must re-upload
Wrong file type (not PDF/image) 400 Error alert — no fallback
Anthropic API error / timeout 500 Warning alert + demo fallback
Network error (fetch fails) n/a .catch() handler → warning alert + demo fallback
Section 08
Security
The parser is designed to keep API credentials server-side and avoid any file persistence.
🔑
API Key Isolation
ANTHROPIC_API_KEY is stored as a Heroku config var. It is never sent to the browser, never referenced in React code, and never committed to git. The Express server reads it at runtime via process.env.ANTHROPIC_API_KEY.
🚫
No File Persistence
Files are held in memory only — multer uses memoryStorage(), never writing to disk. The buffer is passed directly to the Anthropic SDK and discarded after the API call completes. Nothing is retained server-side.
🔒
HTTPS Only
Heroku enforces HTTPS in production. All file uploads and API calls are encrypted in transit. The prototype does not implement additional auth — production would add tenant API key validation on all /api/* routes.
Section 09
Future Extensions
Planned enhancements to the parsing pipeline. Each is independently implementable.
F1
Confidence scores per line
Ask Claude to return a confidence: 0–1 value per extracted line. Flag low-confidence rows (e.g. <0.7) with an amber indicator for manual review before bulk apply.
  • Prompt update
  • UI: amber flag on low-confidence rows
  • Filter: "show only uncertain"
F2
Address / site matching — Tier 4
If a statement line description contains a suburb or postcode, fuzzy-match against the job site address stored on the PO. Resolves statements where the supplier uses a site address instead of a PO reference.
  • Extract address from description
  • Postcode / suburb normalisation
  • Match against po.notes or job address
F3
Haiku fallback for statements
Route statement parsing through claude-haiku-4-5 ($1/$5 per MTok) instead of Opus. Statements have a simpler output schema — Haiku may be sufficient, reducing cost by ~80%. Run A/B test to validate accuracy.
  • Model config env var
  • A/B accuracy test
  • ~$0.01/statement vs $0.05
F4
Prompt caching
Cache the system prompt block using Anthropic's prompt caching (5-min cache at 1.25× write, 0.1× read). At 160 docs/month the saving is small (~$0.003/doc) but significant at scale.
  • Add cache_control to prompt block
  • Monitor cache hit rate
F5
Batch API — overnight processing
Queue 50+ documents overnight via Anthropic's Batch API at a 50% cost discount. Suitable for end-of-month statement runs. Results polled via webhook or cron job.
  • Batch job queue (BullMQ)
  • Webhook / poll for results
  • 50% cost saving on batch
F6
Supplier-specific prompt hints
Store per-supplier column mapping hints (e.g. "for Daikin: PO ref is in column 'Purchase Order No.'"). Inject into the prompt as additional context to improve extraction accuracy for known statement formats.
  • Supplier settings screen
  • Prompt hint injection
  • Format library (Daikin, Reece, etc.)
