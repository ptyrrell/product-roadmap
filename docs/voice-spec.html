Section 01
Overview
The FieldInsight Scheduler allows customers to self-book field service appointments through any of four channels — Voice (VOIP/IVR), Web Chat, SMS, and Email — with a single AI persona (Fiona) managing the conversation and writing confirmed bookings directly into the scheduler.
Prototype status: The current prototype uses the browser's native Web Speech API for voice (STT + TTS). This spec defines the production implementation using real telephony. The AI, booking logic, and multi-channel architecture remain identical across prototype and production.

Prototype vs Production: The current browser prototype uses window.SpeechRecognition (STT) and window.speechSynthesis (TTS) — these are Chrome-only and not suitable for real phone calls. Production replaces these with Twilio's <Gather input="speech"> for STT and <Say> / Amazon Polly for TTS.

How Twilio's Speech IVR Works

// 1. Customer calls Twilio number → Twilio POSTs to /api/voice/incoming
// 2. Backend responds with TwiML to gather speech:

<Response>
  <Say voice="Polly.Joanna">Hi, I'm Fiona from FieldInsight. How can I help?</Say>
  <Gather input="speech" action="/api/voice/process" speechTimeout="auto"
         speechModel="phone_call" language="en-AU">
  </Gather>
</Response>

// 3. Twilio STT converts speech → text → POSTs SpeechResult to /api/voice/process
// 4. Backend sends text + conversation history to Claude
// 5. Claude responds with booking intent + natural language reply
// 6. Backend wraps reply in <Say> + next <Gather> TwiML
// 7. Loop continues until booking confirmed or call ends
Section 03
Voice Call Flow
Step-by-step sequence of a customer calling to book a job, from PSTN entry through to booking confirmation and SMS receipt.
1
Inbound Call Received Twilio
Customer dials the FieldInsight IVR number. Twilio receives the call on the PSTN and immediately POSTs a webhook to the FieldInsight backend with call metadata (CallSid, From, To).
POST /api/voice/incoming Body: CallSid=CA123&From=+61400000000&To=+61391234567
2
Session Initialised + Greeting Played Backend
Backend creates an in-memory conversation session keyed by CallSid. Responds with TwiML containing Fiona's opening greeting and a <Gather> to listen for customer speech. Voice engine: Amazon Polly Joanna (Australian English).
<Response> <Say voice="Polly.Joanna">Hi, I'm Fiona from FieldInsight. How can I help you today?</Say> <Gather input="speech" action="/api/voice/process" speechTimeout="auto" speechModel="phone_call" language="en-AU" /> </Response>
3
Customer Speaks Twilio STT
Twilio's speech recognition model (phone_call) converts the customer's speech to text. The confidence score is also returned. If confidence < 0.4, Fiona asks the customer to repeat.
POST /api/voice/process Body: CallSid=CA123&SpeechResult=I+need+to+book+an+air+con+service&Confidence=0.92
4
Claude Processes Turn Claude API
Backend appends the user turn to the session history and calls Claude with the full conversation. The system prompt includes available time slots, job types, and the VOIP channel style directive ("natural spoken language, no markdown, short sentences").
// Session structure { callSid: "CA123", from: "+61400000000", history: [ { role: "user", content: "I need to book an air con service" }, { role: "assistant", content: "Of course! Could I get your name please?" } ], bookingData: {}, startedAt: "2026-03-23T10:00:00Z" }
5
Response Spoken Back Twilio TTS
Backend strips any markdown from Claude's reply, wraps it in a <Say> TwiML verb, and appends another <Gather> to continue the conversation. The loop continues for up to 15 turns or 8 minutes.
<Response> <Say voice="Polly.Joanna">Great, I have Tuesday the 25th at 9am or Thursday the 27th at 2pm available. Which works best for you?</Say> <Gather input="speech" action="/api/voice/process" speechTimeout="auto" /> </Response>
6
Booking Confirmed + Written Backend
When Claude outputs a BOOKING:{...} JSON line, the backend parses it, writes the job to the scheduler, and plays a confirmation message. The call can then end gracefully.
// Claude output (stripped from spoken response before TTS) BOOKING:{"tech":"Jake Morrison","customer":"David Chen","type":"HVAC", "address":"42 Collins St Melbourne","mobile":"+61412345678", "email":"d.chen@example.com","date":"2026-03-25","startHour":9,"duration":2}
7
Confirmation SMS + Email Sent Twilio SMS
Backend fires a confirmation SMS to the caller's number (extracted from the call's From field — no need to ask if they called from their mobile) and a confirmation email if collected. A reminder is scheduled 1 hour before the job using a cron job or Twilio Scheduled Messages.
// Confirmation SMS (Twilio Messages API) "Your FieldInsight booking is confirmed: HVAC Service, Tue 25 Mar at 9am. 42 Collins St Melbourne. Reply CANCEL to cancel. FieldInsight Team."
8
Call Ends + Session Cleaned Up Backend
Twilio fires a status callback on call completion. Backend persists the session transcript to the job record for internal review, then clears the in-memory session. The booking is now visible in the internal scheduler.
Section 04
Voice Integration Options
There are four ways a FieldInsight customer can connect a real phone number to Fiona's IVR. The right choice depends on their technical comfort level, whether they already have a Twilio account, and how they want to integrate with their existing phone system.
★ Recommended
Option A
FieldInsight Managed Number
We provision a Twilio number on your behalf
FieldInsight holds a Twilio sub-account (or uses Twilio's Marketplace / white-label reseller model). When a business activates VOIP in settings, they're assigned a dedicated local number in their area code instantly. No Twilio account needed. Cost is bundled into the FieldInsight plan or as a small add-on.
  • Zero setup for the business
  • One-click activation
  • We handle all Twilio config
  • Best UX for non-technical customers
  • FieldInsight carries Twilio cost
  • Number is "ours", not theirs
  • Requires Twilio reseller agreement
Business effort:
Minimal
Implementation: Use Twilio's Programmable Voice + sub-account API. Register number → set webhook to /api/voice/incoming?tenant={id} → done. Implemented via Twilio's node.js SDK on first VOIP activation.
Self-managed
Option B
Bring Your Own Twilio
Business supplies Twilio credentials
The business signs up for their own Twilio account, purchases a number, and pastes their Account SID + Auth Token + Phone Number into FieldInsight settings. FieldInsight automatically configures the number's webhook URL to point at the IVR. The business is billed directly by Twilio.
  • Business owns their number
  • Direct Twilio billing
  • Full Twilio console access
  • Works for existing Twilio users
  • Business must create Twilio account
  • 3–5 step setup process
  • Credential management / security
Business effort:
Moderate
// Settings fields required from business: TWILIO_ACCOUNT_SID=ACxxxxxxxxxxxxxxxx TWILIO_AUTH_TOKEN=xxxxxxxxxxxxxxxx TWILIO_FROM_NUMBER=+61391234567 // On save: FieldInsight auto-sets webhook via Twilio REST API
★ Best for SMBs
Option C
Dedicated Redirect Number
Forward any existing number to a FieldInsight IVR line
FieldInsight provides a dedicated national number (e.g. 1300 or local) per tenant. The business then redirects their existing business line to this number — either always, or conditionally (after-hours, on busy, or via an IVR menu option like "Press 2 to book"). No Twilio account, no setup — just a call forward on their existing phone plan.
  • Business keeps their existing number
  • Works with any carrier
  • No Twilio account needed
  • Conditional forward = after-hours only
  • Forwarding costs on their carrier
  • Business must set up forward
  • Slight latency from double-hop
Business effort:
Low — just a call forward
Use case: "Divert your after-hours calls to 03 9123 4567 and Fiona will handle all booking requests while you're closed." This is the most compelling pitch to a small service business.
Advanced
Option D
SIP Trunk / PBX Integration
Direct SIP endpoint for businesses with a PBX
For businesses running RingCentral, 3CX, FreePBX, or any SIP-capable PBX. FieldInsight exposes a SIP URI (via Twilio SIP Domain) that can be configured as an extension or IVR transfer target. The call arrives at FieldInsight directly over SIP, bypassing the PSTN entirely.
  • No carrier call-forward costs
  • Deep PBX integration possible
  • Lower latency
  • Works with existing call flows
  • Requires IT involvement
  • Not suitable for small businesses
  • SIP config complexity
Business effort:
Significant IT setup
// SIP URI assigned per tenant: sip:tenant-abc@fi-ivr.sip.twilio.com // Configure in PBX as outbound route or IVR transfer target
Recommended launch strategy: Launch with Option A (Managed Number) for simplicity and conversion rate. Add Option C (Redirect) as the marketing pitch for after-hours ("never miss a booking"). Add Option B (BYOT) for enterprise/tech-savvy customers in v2. Option D is a v3 enterprise feature.
Section 05
API Endpoints
Full list of backend endpoints required for production voice IVR, plus the existing multi-channel endpoints. All endpoints are authenticated with a tenant API key in production.
MethodEndpointDescriptionUsed By
POST /api/voice/incoming Twilio webhook for inbound calls. Creates session, returns greeting TwiML. Twilio
POST /api/voice/process Receives SpeechResult from Twilio Gather, sends to Claude, returns next TwiML turn. Twilio
POST /api/voice/status Twilio status callback on call completion. Persists transcript, cleans up session. Twilio
POST /api/voice/provision Admin: provision a new Twilio number for a tenant (Option A). Returns assigned number. Backend
POST /api/chat Web/SMS chat turn. Sends user message + history to Claude. Returns reply + booking data. Frontend
GET /api/schedule Returns jobs, techs, techSettings, bookingSettings for the scheduler view. Frontend
POST /api/schedule/jobs Create a confirmed job. Called by booking parser when BOOKING JSON detected. Backend
GET /api/settings/booking Fetch booking rules (buffer hours, working days, hours, lunch break). Frontend
POST /api/settings/booking Update booking rules. Frontend
GET /api/settings/prompt Fetch AI persona + prompt settings (Fiona config). Frontend
GET /api/settings/prompt/preview Returns the compiled system prompt as it will be sent to Claude. Frontend
POST /api/send-sms Send an SMS via Twilio (confirmation, reminder, or manual from Booking Details view). Frontend Backend
POST /api/validate-address Validate + geocode an address via Google Maps Geocoding API. Frontend
GET /api/config Returns public config (Google Maps API key, tenant VOIP number). Frontend

Section 07
Security Considerations

Twilio Webhook Validation

All Twilio webhooks must be validated using the X-Twilio-Signature header to prevent spoofed requests.
import twilio from 'twilio';
const valid = twilio.validateRequest(
  authToken, signature,
  req.url, req.body
);
if (!valid) return res.status(403).end();

Credential Storage

Twilio credentials and API keys must never be stored in frontend code or committed to git.
# .env (never committed)
ANTHROPIC_API_KEY=sk-ant-...
TWILIO_ACCOUNT_SID=ACxxx
TWILIO_AUTH_TOKEN=xxxxxx
TWILIO_FROM_NUMBER=+613...
GOOGLE_MAPS_API_KEY=AIza...

Rate Limiting

Voice sessions: max 15 turns per call (prevents runaway Claude billing). Chat: 60 req/min per IP. SMS: Twilio rate limits apply. Implement using express-rate-limit.

PII Handling

Call transcripts contain PII (name, address, mobile). Store encrypted at rest. Implement transcript retention policy (default: 90 days). Australia: bound by Privacy Act 1988 for customer data.
Section 08
Implementation Roadmap
Phased build plan to move from the current browser-based prototype to production-grade telephony. Each phase is independently deployable.
P1
Production Voice IVR — Core ~3–5 days dev
Replace Web Speech API with Twilio Programmable Voice. Implement /api/voice/incoming + /api/voice/process + session management. Use Amazon Polly Joanna (AU English) for TTS. Connect to existing Claude booking logic.
  • Twilio Voice webhooks
  • In-memory session store (Redis for scale)
  • TwiML builder utility
  • Polly voice config
  • Low-confidence fallback handler
  • 8-minute / 15-turn call guard
P1
Managed Number Provisioning (Option A) ~1–2 days dev
Implement /api/voice/provision using Twilio's IncomingPhoneNumbers API. One-click number assignment per tenant with webhook auto-configuration. Display assigned number in settings.
  • Twilio sub-account per tenant (or shared + tenant routing)
  • Number search by area code
  • Webhook auto-configuration
  • Settings UI: "Activate VOIP Booking Line"
P2
BYOT + Redirect Number (Options B + C) ~2–3 days dev
Add credential input UI for BYOT. Implement auto-webhook-configuration via Twilio REST when BYOT credentials saved. For Option C, provision a shared IVR pool number with tenant routing by To parameter.
  • BYOT credential form + validation
  • Auto-webhook setter via Twilio API
  • Tenant routing by To number
  • Redirect guide / setup wizard
P2
Persistent Database ~3–5 days dev
Move from in-memory storage to PostgreSQL. Jobs, sessions, transcripts, settings all persisted. Multi-tenant isolation. Required before production launch.
  • PostgreSQL schema + migrations
  • Prisma ORM (or Knex)
  • Multi-tenant row-level security
  • Session store → Redis
P3
Scheduled Reminders + Email Confirmations ~2 days dev
Implement job-based reminder scheduler. Send SMS (Twilio) and email (SendGrid) confirmations on booking. Schedule a 1-hour-before reminder using a cron queue (BullMQ + Redis or Heroku Scheduler).
  • BullMQ job queue
  • SendGrid email templates
  • SMS confirmation on booking
  • 1-hour reminder job
  • CANCEL reply handling
P4
SIP Trunk Integration (Option D) + Analytics ~5 days dev
Twilio SIP Domain setup for PBX customers. IVR analytics dashboard (call volume, booking rate, avg call duration, abandonment). Voice A/B testing (alternate prompts).
  • Twilio SIP Domain per tenant
  • SIP credentials + INVITE auth
  • Analytics dashboard
  • Call recording (opt-in)
Stack recommendation: Node.js (ESM) · Express 5 · PostgreSQL + Prisma · Redis (sessions + job queue) · BullMQ (reminders) · Twilio SDK v4 · @anthropic-ai/sdk · SendGrid · Heroku (or Railway for simpler Postgres provisioning). Existing prototype already uses Node.js/Express — P1 is a direct evolution, not a rewrite.
