FI-022 — FieldInsight Public API + OAuth + MCP Server
Expose FieldInsight’s REST API to Alfred, Claude Desktop, Cursor, and any other MCP-aware agent. Standard OAuth 2.0 Authorization Code Grant → secure token store → typed API client → MCP tool layer. Shipped in four phases: Phases 1–3 are CORE, Phase 4 is Nice to Have.
📘 Spec — v0.1 ⏳ Not Started 🟢 OAuth 2.0 Standard Effort: ~2.5 days MVP · ~5 days full 🤖 MCP Ready Owner: Paul Tyrrell

Section 01
Overview
FieldInsight exposes a RESTful Public API secured by OAuth 2.0. Today nothing at Open Claw consumes it — Alfred can’t read jobs, Cursor can’t touch customer records, Claude Desktop can’t answer “which jobs are on today?”. This spec scopes a minimal, secure bridge: an OAuth token service, a typed API client, and an MCP server that exposes FieldInsight as callable tools to any MCP-aware agent.

Section 03

Phase 1 · CORE
OAuth Foundation
Implement the Authorization Code Grant flow against FieldInsight. No MCP, no API calls yet — just: get a token, store it, refresh it.
1
Register app at app.fieldinsight.com/developer/signup
Client type Confidential, grant type Authorization code. Register a stable redirect_uri — HTTPS only in production.
2
Redirect user to authorize URL
Include a random state for CSRF protection, persist it server-side, verify on callback.
GET https://app.fieldinsight.com/oauth/authorize ?response_type=code &client_id=<client_id> &redirect_uri=https://openclaw.app/oauth/fi/callback &state=<csrf_token>
3
Exchange code for tokens at /oauth/token/
POST https://app.fieldinsight.com/oauth/token/ Content-Type: application/x-www-form-urlencoded grant_type=authorization_code &client_id=<client_id> &client_secret=<client_secret> &code=<authorization_code> &redirect_uri=https://openclaw.app/oauth/fi/callback
4
Persist tokens encrypted at rest
Store access_token, refresh_token, expires_at, tenant_id keyed by FieldInsight account. Encrypt with a KMS-managed key — never plaintext in the DB or logs.
5
Auto-refresh when expires_at is within 5 minutes
Background job + on-demand refresh on 401. Access token lives 10 hours — refreshing is rare but must be bulletproof.
  • ✓ Register OAuth app + store client_id / client_secret as env vars
  • ✓ GET /oauth/fi/authorize route — generates state, redirects to FieldInsight
  • ✓ GET /oauth/fi/callback route — validates state, exchanges code
  • ✓ Token store schema: fi_tokens(tenant_id, access_token_enc, refresh_token_enc, expires_at, scope)
  • ✓ Refresh helper: getValidAccessToken(tenantId) — refreshes if <5min from expiry
  • ✓ Error paths: invalid_grant → force re-auth flow with clear UX
  • ✓ Unit tests: token encryption round-trip · refresh logic · state validation
  • ✓ Multi-tenant console page to view connected FI accounts Optional
  • ✓ Revoke endpoint — explicit disconnect Optional
Phase 2 · CORE
Typed API Client
A thin wrapper that takes a tenantId, fetches a valid bearer token from Phase 1, and makes a typed request against https://app.fieldinsight.com/public-api/.
# Python sketch — same pattern in TypeScript
class FieldInsightClient:
    def __init__(self, tenant_id, tokens):
        self.tenant_id = tenant_id
        self.tokens    = tokens
        self.base      = "https://app.fieldinsight.com/public-api"

    async def _request(self, method, path, **kw):
        token = await self.tokens.get_valid_access_token(self.tenant_id)
        headers = {"Authorization": f"Bearer {token}"}
        r = await httpx.request(method, self.base + path, headers=headers, **kw)
        if r.status_code == 401:
            await self.tokens.force_refresh(self.tenant_id)
            token = await self.tokens.get_valid_access_token(self.tenant_id)
            headers["Authorization"] = f"Bearer {token}"
            r = await httpx.request(method, self.base + path, headers=headers, **kw)
        if r.status_code == 429:
            await backoff(r)
        r.raise_for_status()
        return r.json()

    async def get_job(self, job_id): ...
    async def list_customers(self, **filters): ...
    async def list_timesheets(self, **filters): ...
    async def list_technicians(self): ...

Phase 3 · CORE
MCP Server — Read-Only Tools
A standalone MCP server exposing six read-only tools. Ships as fi-mcp — users install it, point it at their token store, and it’s available in Claude Desktop, Cursor, Alfred, etc.
fieldinsight_list_jobs
List jobs with filters: status, date_from, date_to, technician_id, site_id. Returns paginated typed results.
fieldinsight_get_job
Fetch a single job by ID with full detail — line items, technician, status history, attached notes.
fieldinsight_list_customers
Search customers by name, email, phone. Returns core contact + billing fields.
fieldinsight_get_customer
Full customer record including sites, contacts, assets, and recent job history.
fieldinsight_list_timesheets
Timesheets filtered by technician + date range. Returns total hours, job linkage, and billing status.
fieldinsight_list_technicians
Active technicians with role, skills, and current assignment count.
  • ✓ MCP manifest fi-mcp.json — name, description, tool schemas
  • ✓ Six tool handlers wiring MCP args → Phase 2 client calls
  • ✓ Input validation + friendly error messages for agents
  • ✓ stdio transport for Claude Desktop / Cursor
  • ✓ README.md with install + config steps
  • ✓ End-to-end test: from Cursor, ask “what jobs are open for tech X this week?” and verify real data
  • ✓ HTTP transport variant for remote deployment Optional
  • ✓ Publish to MCP registry Optional
