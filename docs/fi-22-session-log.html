Section 03
Key Decisions
Decisions taken during the thread that change how this ticket is scoped or built. Recorded as Q → A → because, so future-me can reconstruct the reasoning.
Decision 01 · Scope
Use four phases — Phases 1–3 CORE, Phase 4 Nice-to-Have
Phases are not independently shippable until all three CORE phases land. Phase 4 starts only after CORE is in use.
Because: avoids the “fake MVP” trap where an early phase ships but no human can use the result.
Decision 02 · Tagging
OPTIONAL items live inside a phase, not as their own phase
If an item can be dropped without breaking the phase’s Definition of Done, it gets the OPTIONAL tag. If it can’t be dropped, it’s CORE.
Because: keeps scope creep visible without turning every “nice idea” into a new phase.
Decision 03 · Auth
Backend OAuth service only — no PKCE / no SPA flow
FieldInsight’s OAuth implementation is a confidential-client Authorization Code Grant. No mention of PKCE in the docs.
Because: server-side only is the documented happy path; SPA / mobile would have to proxy through our backend anyway.
Decision 04 · Tokens
Encrypted multi-tenant token store, refresh ≥5 min before expiry
10-hour access tokens stored encrypted at rest, keyed by tenant_id. A background refresher fires when expires_at - now < 300s; on-demand refresh on 401.
Because: 10 hours is generous, but never let a token expire mid-call from an agent.
Decision 05 · MCP surface
14 read tools across 4 categories — full operational coverage
Jobs & Customers (4) · Operations (4) · Financial (4) · Catalogue (2). Effort recalibrated to 2–3 days for Phase 3.
Because: a partial read surface forces the agent to fall back to web UI for half its questions — defeats the point.
Decision 06 · Write safety
All write tools require confirm=true at the MCP layer
No mutating call goes through without an explicit confirmation flag from the agent caller. Phase 4 only.
Because: agents hallucinate. A second-factor check at the tool boundary is cheaper than reversing a botched job creation.
Decision 07 · Sub-tickets
FI-022 gets 8 sub-tickets (FI-022.1 … FI-022.8)
Sub-tickets live in the spec, not on the roadmap. The roadmap card stays as the umbrella. Sub-tickets are picked up as standalone units when scheduled.
Because: keeps the roadmap board scannable while preserving the work-breakdown for execution.
Decision 08 · Auth perimeter
Anyone at @fieldinsight.com can access The Meridian — no per-user allowlist
Confirmed in meridian-auth.js: domain check + Google hosted_domain hint. Vadim and any future Workspace user gain access automatically.
Because: per-user allowlists become stale; Workspace membership is the right perimeter.
